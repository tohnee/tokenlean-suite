# Headroom 的定位 + 缓存时长:两个深入问题的回答

> 本文回答两个问题:① Headroom 在"五层蛋糕"里属于哪一层、它的 CCR 机制是否被现有模型覆盖;② 各种缓存的时长前提到底是多久。两者都有可运行实现支撑(`cache-ttl.mjs`,20 测试通过)。

---

## 问题一:Headroom 属于蛋糕哪一层?CCR 是不是一个新维度?

### Headroom 不是"一层",是一个跨层的复合管线

深入读了 Headroom 的架构后,结论很明确:**Headroom 不属于单一层,它是一条把多层串起来的流水线**。它的各个组件精确地落在不同层:

| Headroom 组件 | 做什么 | 落在蛋糕哪一层 |
|---|---|---|
| **CacheAligner** | 检测 system prompt 里的时间戳/UUID/session token,移到末尾,稳定前缀让 KV 缓存命中 | **L1 输入缓存** —— 正是我 `prompt-assembler` 做的事(前缀稳定化) |
| **SmartCrusher** | JSON 智能压缩(保留 error/离群值/BM25 命中项 + 首尾,其余本地缓存) | **L2 未来输入** —— 压 tool 输出 |
| **CodeCompressor** | tree-sitter AST 压缩代码(保 import/签名/类型,保证语法有效) | **L2 未来输入** —— 压代码 |
| **Kompress-base** | 在 agent trace 上训练的 HF 小模型,压自然语言/日志 | **L2 未来输入** —— 压 prose |
| **CCR(Compress-Cache-Retrieve)** | 压缩时把原文存本地 SQLite,注入 `[hash=abc]` 标记 + `headroom_retrieve` 工具,模型需要时按 hash(可带 BM25 query)取回原文 | **← 这是现有五层没明确覆盖的机制** |

所以前四个组件,我的模型都覆盖了:CacheAligner = L1(和 prompt-assembler 重叠),三个 Compressor = L2(和"未来输入压缩"重叠,Headroom 做得更专业)。**唯独 CCR 不属于现有任何一层。**

### CCR 为什么是一个新维度:可逆压缩 / 按需检索

CCR 的本质不是"压缩",也不是"路由",而是**改变了压缩的信息论性质**:

```
传统压缩(L2):      有损。20 行喂给模型,另外 980 行永久丢弃 → 省 token 但有信息损失风险
CCR:               把"有损呈现"和"无损保留"解耦。
                   模型只看到 20 行(省 token),原始 1000 行存本地;
                   模型若判断不够,调 headroom_retrieve(hash, query) 按需取回 → 信息零丢失
```

这消除了压缩的经典权衡(省 token vs 保信息)。它的关键特征:
- **呈现是有损的,存储是无损的** —— 给模型的上下文小,但数据没丢
- **检索由模型决定,不是预先决定** —— 模型自己判断 20 行够不够,不够才取回
- **检索是带查询的** —— `headroom_retrieve(hash, query)` 在本地缓存里 BM25 搜,只取回相关子集
- **对客户端透明** —— 应用层看不到这些往返

这既不是单纯的 L2 压缩(它可逆、可检索),也不是 L5 路由(它不换模型)。它是一个**正交的新机制**:把"上下文"从"一次性塞满"变成"按需拉取的可寻址存储"。

### 现有模型没覆盖,补进去:把它叫做"L2.5 可逆压缩 / 按需上下文"

我决定不硬塞进现有五层,而是把它明确为一个**横切机制**,因为它可以套在 L2 之上(压缩任何东西时都让它可逆),也改变了 L2 的风险特性。在更新后的综述里,它作为 L2 的一个"升级形态"明确标出,并指出本项目 MCP 的 `fs_read_hashed`(读文件给 hash 锚点、模型按锚点取具体行)其实是**同一思想的一个轻量特例**——锚点就是 hash,按需读就是 retrieve,只是没做本地原文缓存。这让我的项目和 CCR 的关系清楚了:**同源思想,Headroom 做成了通用的可逆压缩层,本项目只在文件编辑场景做了它的一个特例。**

### 对本项目的影响

- `prompt-assembler` 的 CacheAligner 部分与 Headroom 重叠 → 二者择一(自建 RAG 服务用 assembler 库,过 proxy 的 agent 流量用 Headroom)。
- L2 压缩本项目本来就建议用 Headroom(不重复造轮子),现在更明确:**Headroom 的真正独特价值是 CCR**,不是压缩率数字。选 Headroom 主要为了 CCR 的可逆性。
- 本项目 MCP 的 hash 读取可以演进:给 `fs_read_hashed` 加一个本地原文缓存 + `retrieve_by_hash`,就成了 CCR 的文件场景完整版。这是一个明确的、有价值的待办。

## 问题二:缓存的时长前提——不能无限缓存全部历史

这是一个被普遍忽略却很关键的约束。答案分三层:**有明确的 TTL、只缓存前缀不缓存全部历史、即使能缓存也受上下文窗口约束。**

### 各家缓存的确切时长(已联网核实,2026-06)

| 提供商 | 默认 TTL | 可延长到 | 命中是否刷新 TTL | 读价 | 关键特性 |
|---|---|---|---|---|---|
| **Anthropic** | **5 分钟** | 1 小时(`ttl:"1h"`,写价 2×) | ✅ 每次命中重置 5 分钟 | 0.1× | 活跃会话只要间隔 < TTL 就一直热;静默超时则静默重写 |
| **OpenAI** | **5–10 分钟**(不活跃) | 24 小时(`prompt_cache_retention:"24h"`,GPT-5.x/4.1) | ✅ 活跃保温 | 0.25–0.5× | 写免费,所以 miss 的代价只是延迟+那一次全价 |
| **DeepSeek** | **小时级(磁盘)** | — | ✅ | **0.02×(省98%)** | KV 存分布式磁盘(MLA 压缩使其可行),对静默间隔最宽容 |
| **Gemini** | **1 小时**(显式缓存,可自定义) | 自定义 | ❌ 按 token-hour **付存储费** | 0.25× | 唯一"缓存越久标准成本越高"的——空闲也付费 |

核心事实:**默认都是分钟级**。Anthropic/OpenAI 默认 5 分钟,只有 DeepSeek(磁盘)和 Gemini(付费)能到小时级。

### 为什么不能"无限缓存全部历史":三个约束力

1. **TTL 驱逐**:分钟级默认窗口。安静超过 TTL 的间隔就丢缓存,下次调用静默重写(再付一次写溢价)。
2. **只缓存前缀,不缓存任意历史**:只有 byte 稳定的**前缀**会缓存。断点之后任何会变/重排的内容永远不是被缓存的部分。所以你缓存的是稳定的头,不是不断增长的尾。
3. **上下文窗口 + 成本上限**:就算缓存永不过期,每个被缓存的 token **仍然每轮被重新发送、重新计费(按 0.1×)、并且占用上下文窗口**。超过窗口约 12% 后质量下降("context rot")。所以**无论缓存如何,历史都必须被压缩/裁剪**——缓存让"重发"变便宜,但不会变免费,也永远不会让它在窗口里变得没有重量。

第 3 点是最深的认知:**缓存和压缩不是二选一,而是必须同时做。** 缓存解决"重发的单价",压缩/compaction 解决"重发的数量和窗口占用"。只做缓存不做压缩,会撞上 context rot;只做压缩不做缓存,每次压缩都重写前缀、破坏缓存。两者的协调点就是:**低频压缩 + 稳定前缀**(Reasonix 的核心策略)。

### 实操:多久该用哪个 TTL 档

`cache-ttl.mjs` 的 `recommendTtl()` 把这个决策做成可计算的(20 测试验证):

| 你的请求平均间隔 | Anthropic 建议 | 理由 |
|---|---|---|
| < 5 分钟 | 默认 5min(命中自动续期) | 活跃流量自己保温,无需额外成本 |
| 5 分钟 – 1 小时 | 显式 `ttl:"1h"` | 超过默认窗口;1h 写价 2×,但前缀被读 ≥2 次就回本 |
| > 1 小时 | 缓存帮不上 | 连扩展窗口都超了,每次冷 miss。要么把请求批得更密,要么接受全价 |
| 任意(DeepSeek) | 默认即可 | 磁盘缓存小时级,最宽容 |

**一个常被忽视的坑**:Anthropic 2026-03 把默认 TTL 从 1h 静默改成 5min,导致稀疏使用的订阅用户命中率暴跌。结论:**生产环境永远显式声明 TTL,不依赖默认值。**

### 代码

```bash
node cli.mjs ttl                 # 三个约束力 + 各家时长总览

```


## 两个问题的合并结论

- **Headroom 是跨层管线**:CacheAligner→L1、三个 Compressor→L2、**CCR 是一个新机制**(可逆压缩/按需检索),现有五层没明确覆盖,已补为 L2 的升级形态;本项目 MCP 的 hash 读取是同源思想的文件场景特例。
- **缓存有硬时长**:默认分钟级(Anthropic/OpenAI 5 分钟),DeepSeek 磁盘小时级,Gemini 付费可调。不能无限缓存全部历史,受 TTL、前缀限定、上下文窗口三重约束。**缓存与压缩必须同时做**,协调点是"低频压缩 + 稳定前缀"。
