Token Optimization · 系统综述

省 Token 不是一件事,
是五层叠起来的一块蛋糕。

把 coding agent 的 token 账单拆开看,会发现它来自互相独立的来源。每一层有自己的物理原理、自己的最优工具、自己的天花板。混为一谈,是大多数"省 token"努力收效甚微的根因。

受众 · coding agent / copilot 的搭建者与重度用户 主张 · 分层施治,按层选工具,叠加而非堆砌 基准 · 本项目数字可复现;生态数字标注来源
The Five-Layer Cake

五层蛋糕模型

从最贵、最稳定的底层(输入缓存)到最灵活的顶层(任务路由)。越往下越靠近模型推理的物理事实,越往上越靠近行为与调度。每一层都标注了它属于哪个 token 维度。

L1输入缓存层INPUT · prefix / prompt / KV cache

每轮请求里输入给模型的字节。命中 prefix cache 后按约 0.1× 基价计费。三个名字其实是同一底层机制(跨请求 KV 复用)的不同切面:KV cache 在单请求内、prefix cache 跨请求、prompt cache 是它的商业 API 包装。命中的唯一条件是 byte-exact 前缀稳定。

prefix cachingprompt cacheKV cacheTTL 1h vs 5mincache_control 断点
物理事实:输入单价低,但长会话里输入线性增长,缓存是这一层唯一的杠杆。错误的压缩会破坏它。
L2未来输入层FUTURE INPUT · history bloat

大工具输出一旦进入对话历史,会在后续每一轮被重复计费。第 3 轮 cat 的 500 行日志,会被发送到第 4…N 轮。这是长会话里最大的、也最容易被忽视的成本来源。优化手段:输出预算、压缩、scratch 隔离、按需检索替代全量读。

tool output budgetcompactionscratch 隔离code graph 检索CLI 输出压缩CCR 可逆压缩 ★
杠杆最大的一层:单次省下的量会乘以"它本该重复出现的轮数"。但压缩历史 = 改写前缀 = 可能伤到 L1。
升级形态(L2.5)——可逆压缩 / 按需检索:Headroom 的 CCR 把"有损呈现"与"无损保留"解耦——只给模型 20 行、原文 1000 行存本地,模型不够再按 hash(可带查询)取回。消除了"省 token vs 保信息"的经典权衡。本项目 MCP 的 hash 读取是同源思想的文件场景特例。
L3输出·结构层OUTPUT (structured) · edits & tool calls

模型生成的结构化输出——主要是文件编辑指令。输出 token 单价是输入的 4-5 倍。手段:用紧凑的编辑格式(targeted Edit / hash 锚点)替代全文件重写,不复述未改内容。诚实提醒:对已经会写紧凑 diff 的强模型,这一层省不了多少 token;真正的收益是编辑安全与弱模型可靠性。

targeted edithash-anchored editno full rewritefail-fast 安全
vs 全量重写:省约 86%。vs 称职的 native Edit:≈ 0(价值在安全,不在 token)。
L4输出·风格层OUTPUT (prose) · response style

模型生成的自然语言部分——解释、叙述、铺垫。与 L3 正交:L3 管"编辑指令多紧凑",L4 管"话多不多"。手段:约束输出风格,砍掉叙述性废话,保留代码与技术结论。有研究指出,约束模型简短作答在某些基准上反而提升准确率。

terse outputno narrativecaveman 风格style compression
代表工具 caveman 宣称输出 token -65%(峰值 87%);只动"嘴",不动"脑"(推理 token 不变)。
L5任务路由层ROUTING · model selection

最顶层,也是最被低估的:不是省单次请求的 token,而是把请求送到对的模型。简单/机械的任务路由到便宜或本地小模型,只有难任务才用最贵的大模型。难易问题路由把"每 token 单价"这个维度也变成可优化对象。

difficulty routingcheap/local modelclaude-code-routerspeculative / draft
改变的不是 token 数量,而是 token 单价。与下面四层完全正交,纯叠加。
读法 → 越靠底层越接近"推理的物理事实"(缓存命中是引擎层的硬约束),干预它需要越底层的手段(网关/harness);越靠顶层越接近"行为与调度",一个 skill 或一行配置就能动。这也解释了为什么纯 skill 只能碰到 L3-L5,够不到 L1。
Inside Layer 1 · 缓存的真相

输入缓存层,拆开看

L1 是最省钱也最反直觉的一层。四个常被忽略的问题决定了你到底能不能拿到这层的收益。每一条都有可运行、已测试的实现支撑。

Q1单租户 vs 跨用户

商业 prompt cache 按组织/workspace 隔离,不跨租户共享。你蹭不到别人发过的同样 system prompt。

引擎层确实有跨请求的 KV page 共享(PagedAttention / RadixAttention),但 API 出于隔离不暴露给你。

你能控制的,只有自己请求流里的前缀复用——同一 system/tool/KB 在你连续的请求间字节不变地复用。这才是优化对象。

Q2长 system / tool / 历史

按稳定性分四级排进前缀:STATIC(system、tool 定义)→ SESSION(项目上下文、pinned KB)→ ROLLING(历史,append-only)→ VOLATILE(检索结果、问题)。

失效是层级传染的(tools→system→messages),越稳越靠前。历史能进前缀的前提是只追加、不重排。

tool 定义是隐形杀手:加一个工具就让所有用工具的请求缓存失效。按需加载工具(ToolSearch)曾把 DeepSeek 命中率从 98% 打到 81%。

Q3拼装结构 = 成本决策

由用户输入、工具输出、上下文动态拼装时,拼到哪个位置、按什么字节,直接决定命中。三个反模式:

① 可变内容前置(问题/检索结果拼在 system 前)② 前缀内嵌时间戳/UUID(一个日期 token 让全天零命中)③ 非确定性序列化(JSON key 乱序、score 拼进文本)。

本项目把拼装结构做成显式、可检视的产物:排序 + 断点 + 泄漏诊断。

Q4RAG / llm-wiki 命中

最尖锐的问题:top-k 检索每次都变,naive 做法拼在前面 → 前缀每次变 → 零命中。

两手修复:① 钉一个稳定前缀层(KB 索引 / wiki 导航层),放断点后。② 归一化 volatile 尾部:检索结果按 id 排序(不按 score)、剥离 score/rank/timestamp、去重 → 同一组 chunk 产生字节一致的文本。

修复的本质是顺序 + 归一化,不是压缩:发给模型的字节没少,成本降到一折。

How long does cache live

缓存的时长前提:不能无限缓存全部历史

L1 的省钱有一个硬约束常被忽略——缓存有 TTL,而且默认是分钟级。你保温的不是"全部历史",而是一个被流量刷新、限定在窗口内的稳定前缀。数字已联网核实(2026-06)。

提供商默认 TTL可延长到命中续期读价关键特性

OpenAI5–10 分钟24 小时 (retention:"24h")✅ 活跃保温0.25–0.5×写免费 → miss 代价只是延迟+那次全价

FORCE 1

TTL 驱逐

分钟级默认窗口。安静超过 TTL 就丢,下次静默重写、再付一次写溢价。只有 DeepSeek(磁盘)和 Gemini(付费)到小时级。

FORCE 2

只缓存前缀

只有 byte 稳定的前缀会缓存,不是任意历史。断点后任何变动/重排的内容永远不是被缓存的部分。你缓存的是稳定的头,不是增长的尾。

FORCE 3

窗口 + 成本上限

就算永不过期,每个缓存 token 仍每轮重发、按 0.1× 重计费、占用窗口。超过窗口 ~12% 质量下降(context rot)。所以历史无论如何都得压缩。

最深的点 → 缓存和压缩不是二选一,必须同时做:缓存解决"重发的单价",压缩解决"重发的数量和窗口占用"。只缓存不压缩会撞 context rot;只压缩不缓存,每次压缩都重写前缀破坏缓存。协调点是低频压缩 + 稳定前缀。生产环境永远显式声明 TTL——Anthropic 2026-03 把默认从 1h 静默改 5min,稀疏用户命中率暴跌。

Stacking Laws

叠加三铁律

"装得越多省得越多"是错的。能不能叠、叠了是否真有边际收益,由这三条决定。

01

同维同层只能择一

rtk 和本项目 bash-guard 都在 L2 拦 Bash,同时装会争抢改写同一条命令。择最强的(rtk),不要叠。

02

不同维度乘法叠加

L4 砍叙述 × L3 紧凑编辑 × L2 压输出 × L1 命中缓存,作用于不同 token 子集,收益相乘。这才是"五层蛋糕"成立的前提。

03

压缩层只应有一个

压缩历史 = 改写前缀 = 破坏 L1 缓存。两层压缩第二层几乎无边际收益还二次伤缓存。选一个带 CacheAligner 的压缩器,别叠第二个。

Recommended Stack

务实推荐:每维度一个最强工具

按"每层选一个最强、避免同层冲突、压缩器只留一个"组合。本项目的独特位置不在"压得最狠",而在安全 + 预防 + 可观测。

诚实边界
  • 叠加上限受铁律3 限制。 各家宣称相乘理论可达 -85~92%(与公开的"5工具堆叠90%+"一致),但压缩类边际递减,真实落地通常 -70~85%。
  • L3 的 token 节省被普遍高估。 hash 锚点编辑对称职模型几乎不省 token,主要价值是安全与弱模型可靠性。任何宣称"编辑省 40-60%"的,基本是拿"全量重写"当稻草人基线。
  • 采用率是最大的未知数。 工具做得再好,模型不调用等于零。多数 L2/L3 工具要靠权限 deny 原生工具来强制,否则采用率不保证——这一点几乎没人实测过。
  • 压缩对任务质量的真实影响需长期实测。 "95%准确率保持"是作者基准,不等于你的代码库上的表现。激进压缩在复杂调试上可能丢关键信息。
  • 先量基线,再谈优化。 不先用 ccusage / 本项目 hit-rate 量出当前命中率与成本,所有百分比都是别人的数字,不是你的。
一句话:省 token 是五个正交的目标,不是一个。

按层施治——L1 缓存命中、L2 未来输入、L3 编辑结构、L4 输出风格、L5 任务路由——每层选一个最强工具,遵守叠加三铁律,先量基线再优化。本项目(TokenLean)在这套体系里负责 L3 的编辑安全、L1 的预防性审计、以及全局压缩纪律;其余各层与成熟生态工具组合即可。
